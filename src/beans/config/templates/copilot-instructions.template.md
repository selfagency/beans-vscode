# Beans Task Management Rules

<CRITICALLY_IMPORTANT>

1. **Never start work without a bean.** Before writing any code or making any change, find a relevant existing bean or create a new one. Set its status to `in-progress`. There are no exceptions.

2. **Always use beans instead of TodoWrite or ad-hoc todo lists.** Track all work in the bean body, especially a `## Todo` checklist. Never use scratch-pad task lists as the source of truth.

3. **Always use the extension or MCP first.** Invoke Beans operations through the VS Code extension commands or MCP tools. Use the `@beans` chat participant for guidance. Fall back to the CLI only when the extension, chat, and MCP are genuinely unavailable.

4. **Commit beans promptly.**
   - Commit a bean immediately after creating or modifying it — unless you are creating a milestone/epic and are about to create its child beans, in which case commit everything together once the hierarchy is complete.
   - Completed todo items may be committed together with the code changes they relate to.
   - When bean content changed alongside code, include both in the same commit.

5. **Start or resume on the correct issue branch before writing code.**
   - If the issue does not already have a branch, create one when you begin work.
   - If you are resuming work, checkout the existing branch for that issue first.
   - Branch names must follow Git Flow topic branch conventions aligned to bean type and urgency.
   - Use `feature/` for `feature` beans and most `task` work.
   - Use `bugfix/` for normal `bug` work; use `hotfix/` only for urgent production or release-blocking bugs.
   - Use `support/` for milestone/release-support work when a coordination branch is truly needed.
   - `develop` is a base branch, not a per-bean topic branch.
   - Examples: `feature/beans-vscode-1234-add-search`, `bugfix/beans-vscode-987-fix-init-crash`, `hotfix/beans-vscode-321-patch-login-outage`.
   - Push the branch and record it in the bean frontmatter as soon as it exists.

6. **Record the branch and PR in the bean.** As soon as a branch or PR exists, add it to the bean's YAML frontmatter:

   ```yaml
   branch: <git-flow-prefix>/<bean-id>-<slug>
   pr: <pr-number>
   ```

   Commit this update immediately.

7. **Closing a bean.**
   - Completed: add a `## Summary of Changes` section, then set status to `completed`.
   - Scrapped: add a `## Reasons for Scrapping` section, then set status to `scrapped`.
   - Offer follow-up beans for important deferred work.

</CRITICALLY_IMPORTANT>

## Beans Usage Guide for Agents (Extension-First)

This file is generated by the Beans VS Code extension and defines the preferred execution model.

## Interface priority (highest → lowest)

**Always try each level in order. Only descend when the level above is genuinely unavailable.**

1. **Beans VS Code extension commands/UI** ← start here every time
2. **Beans chat participant (`@beans`) and slash commands**
3. **Beans MCP tools**
4. **Beans CLI** ← last resort only; see CLI constraints below

The CLI is a fallback for environments where the extension, chat participant, and MCP are all inaccessible. Do not prefer it for convenience.

## Core Beans workflow

1. **Before any work:** search for a relevant bean using `beans.search`, the tree/search UI, or `@beans /search`. If none fits, create one and set it to `in-progress`.
2. **Create or checkout the issue branch** before coding.
   - If starting new issue work, create a Git Flow topic branch aligned to the bean type.
   - If resuming, checkout the issue's existing branch.
   - Use `feature/` for `feature` and most `task` beans.
   - Use `bugfix/` for normal `bug` beans and `hotfix/` for urgent production bugs.
   - Use `support/` for milestone/release-support work when needed.
   - `develop` is a shared base branch, not a topic branch.
3. **Push branch and add branch/PR to bean frontmatter** as soon as they exist. Commit the bean update.
4. **Maintain a `## Todo` checklist** in the bean body. Update it after every completed step and commit the updated bean (alone or together with related code changes).
5. **On completion**, add `## Summary of Changes`, set status to `completed`, and commit the final bean state.
6. **On scrap**, add `## Reasons for Scrapping`, set status to `scrapped`.

## Finding work

- Use the extension search and tree views first.
- Use `@beans /next`, `/priority`, `/stale`, or `/search` for conversational triage.
- Use `beans_query` with `operation: "ready"` to find actionable unblocked work programmatically.
- Use `beans_view` to inspect full bean details before starting.

## MCP workflow

1. **Read/context first**: prefer `beans_query` with `operation: "llm_context"`, `refresh`, `filter`, `search`, `sort`, `ready`, or `open_config`; use `beans_view` when you need full bean details.
2. **Default sequence**: discover with `beans_query`, inspect with `beans_view`, mutate with `beans_update`, use `beans_complete_tasks` for checklist work, then archive only when explicitly requested.
3. **Mutate only with explicit intent**: prefer `beans_update` for metadata/body changes, `beans_create` for single-bean creation, and `beans_bulk_create` / `beans_bulk_update` when creating or updating batches.
4. **Destructive actions** (delete, archive) require explicit user confirmation unless the user already requested it.
5. Use `beans_bean_file` for direct `.beans` file reads/edits and `beans_output` for logs/troubleshooting instead of ad-hoc file access.
6. MCP is the preferred programmatic interface when the extension UI is not in focus.

### MCP public tool notes

- `beans_update` is the main mutation tool. It supports `status`, `type`, `priority`, `parent`, `clearParent`, `blocking`, `blockedBy`, `body`, `bodyAppend`, `bodyReplace`, and optional optimistic concurrency via `ifMatch`.
- `beans_update` rejects combining `body` with `bodyAppend` or `bodyReplace` in one request.
- `beans_create` prefers `body`; `description` is accepted only as a deprecated alias.
- `beans_bulk_create` and `beans_bulk_update` are best-effort batch tools: expect per-item success/error results instead of one atomic success/failure.
- `beans_query` can generate Copilot instructions via `llm_context`; when it writes an instructions artifact itself, the upstream MCP server writes `.github/instructions/beans-prime.instructions.md`.
- `beans_bean_file` accepts either `some-bean.md` or `.beans/some-bean.md`; the `.beans/` prefix is normalized automatically.
- Use `ifMatch` with the current bean `etag` when concurrent edits are possible.

## Planning mode for epic decomposition

When planning an epic:

1. Confirm goal, constraints, and definition of done.
2. Propose a child-issue map grouped by outcomes.
3. For each child issue, include title, type, priority, and dependencies.
4. Ask for approval before creating any issues.
5. Create all approved child issues and link them with parent relationships.
6. **Commit the parent epic and all child beans together** once the hierarchy is complete.
7. Return created IDs and suggest the first issue to start.

Use compact planning checklist format:

- [ ] `<title>` — type=`<type>`, priority=`<priority>`, depends_on=`<ids|none>`

## CLI fallback (only when extension, chat, and MCP are all unavailable)

**Constraints when using the CLI:**

- Always use `--json` for output; never parse plain-text output.
- Never pass large blocks of body text directly as a shell argument; use GraphQL variables or pipe the query into `beans graphql`.
- Run one operation at a time; do not chain destructive commands.
- Before using the CLI at all, verify that the extension, chat participant, and MCP tooling are genuinely unavailable.

**Allowed CLI command:**

- `beans graphql --json "<query>" [--variables <json>]`
- `beans archive <id>` (only when explicitly requested by the user)
